一次Autorelease实践

发表于 2018-06-25

和朋友在一次谈论autorelease对象释放时机的时候发现了有趣的问题，通常在说到autorelease我们都能看到下面这样一段话：

App启动后，苹果在主线程 RunLoop 里注册了两个 Observer，其回调都是 _wrapRunLoopWithAutoreleasePoolHandler()。
第一个 Observer 监视的事件是 Entry(即将进入Loop)，其回调内会调用 _objc_autoreleasePoolPush() 创建自动释放池。其 order 是-2147483647，优先级最高，保证创建释放池发生在其他所有回调之前。
第二个 Observer 监视了两个事件： BeforeWaiting(准备进入休眠) 时调用_objc_autoreleasePoolPop() 和 _objc_autoreleasePoolPush() 释放旧的池并创建新池；Exit(即将退出Loop) 时调用 _objc_autoreleasePoolPop() 来释放自动释放池。这个 Observer 的 order 是 2147483647，优先级最低，保证其释放池子发生在其他所有回调之后。
在主线程执行的代码，通常是写在诸如事件回调、Timer回调内的。这些回调会被 RunLoop 创建好的 AutoreleasePool 环绕着，所以不会出现内存泄漏，开发者也不必显示创建 Pool 了。

所以就有两个问题：

  1. 为什么会在viewDidAppear:输出为null呢？
  2. 为什么不同字符串输出结果不同呢？

为什么会在viewDidAppear:输出为null。是因为 ViewController 在 loadView 之后便 add 到了 window 层级上，所以 viewDidLoad 和 viewWillAppear 是在同一个 runloop 调用的。然后 viewDidAppear 方法是在动画执行完成后触发的，是因为CoreAnimation 在 RunLoop 中注册了一个 Observer，监听了 BeforeWaiting 和 Exit 事件，所以 CoreAnimation 的执行，是在 BeforeWaiting 和 Exit 时才执行，此时 Runloop 迭代结束，Pop 当前的自动释放池，refrence对象释放，并且在测试了当push animated = NO，viewWillAppear、viewDidAppear 就会在同一个 Runloop 周期内。

同时也可以通过UIView动画来做测试:
[UIView transitionWithView:self.view duration:0.25 options:UIViewAnimationOptionLayoutSubviews animations:^{
NSLog(@”%@”, refrence); log: testtesttest
} completion:^(BOOL finished) {
NSLog(@”%@”, refrence); //log: null
}];

那么为什么不同字符串输出结果不同呢，通过断点能看到 test 的类型为 NSTaggedPointerString，想知道为什么，就得对这个类型做一下了解。
苹果在64位的环境下对NSString，NSNumber等对象做了一些内存优化，所以引入了TaggerPoint，简单的来说就是在64位系统的情况下，指针变量的大小为8字节可以容纳一些 长度较小的内容，所以苹果使用了taggerPoint这种的方式来优化数据存储方式。所以对于NSString来讲，当非字面值常量的数字，英文字母字符串的长度小于等于 9 的时候会自动成为 NSTaggedPointerString 类型，并且查看TaggerPoint的引用计数可以发现为9223372036854775807 这么大的值，所以也可以看出他是不会释放的。这也就说明了为什么两种长度一样的字符串会输出结果不同。
